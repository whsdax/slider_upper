如何用 AI 从零构建 slider_upper V5.2

资深 AI Agent 工程师视角 · 新手对话指南 · 2026-06-16

开篇：AI 协作的正确预期

AI（如 Cursor、ChatGPT、Claude）是结对工程师，不是一键生成器。它能快速写代码、查错、重构，但无法代替你提供硬件事实和验收标准。

本项目交给 AI 之前，你至少需要准备

  • 硬件：PEAK PCAN-USB 适配器、LA 系列推杆、已安装 PCAN 驱动
  • 协议文档：KAI执行器通信32字节协议.xlsx（厂商提供，仓库内不包含）
  • 设备参数：关节 ID（1~10）、推杆型号与行程（如 LA5000 = 90mm）
  • 功能清单：你要什么（校准？移动？GUI？），按优先级排序
  • 验收方式：怎样算「做完了」（例如：校准后误差 < 0.05mm）
工程师经验：协议 xlsx 越早交给 AI 越好。没有它，AI 只能根据零散代码「猜」字节偏移，极易写出能跑但单位错误的解析逻辑。

核心问题解答

Q1：应该先搭主功能，再慢慢完善吗？

是。推荐「分层迭代」——每一层都能独立运行、在真机上验证，再叠下一层。本仓库的真实演进路径就是这样：

flowchart LR P0[P0 can.py硬件层] --> P1[P1 position_light] P1 --> P2[P2 calib_v3/v4] P2 --> P3[P3 slider_upper GUI] P3 --> P4[P4 注释与文档]

不要一上来就说「帮我写个完整上位机」。那会让 AI 同时处理驱动、协议、GUI、线程，返工率极高。

Q2：KAI执行器通信32字节协议.xlsx 需要吗？

需要，且应在 P0/P1 阶段就附上。该 Excel 是 CAN 通信的权威定义：

Sheet内容对应代码
Sheet2SDO 指令（0x600+id）：使能、上报、offset、改 ID 等calib_v3.py、slider_upper._calibration_process
Sheet3TxPDO4 反馈（0x480+id）32 字节字段定义slider_upper._update_status_display
Sheet4故障码等级与错误字状态面板 errorcode 解读

仓库不上传该 xlsx（厂商文档），但你在与 AI 对话时应作为附件或截图提供。

推荐迭代路线图

阶段产物文件目标硬件验收
P0PCANBasic.py + can.pyCAN-FD 能收发PCAN 视图或脚本收到任意报文
P1position_light.py使能→移动→失能推杆按输入长度伸出
P2calib_v3.py → calib_v4.py位置 offset 校准校准后反馈位置接近真实值
P3slider_upper.pyGUI + 状态监控连接、校准、移动、读状态均正常
P4注释、teach.html、slider_upper.spec可维护、可打包交付他人能读懂、能打出 exe
原则：未通过当前阶段验收，不要进入下一阶段。否则 bug 会层层叠加，AI 也难以定位根因。

分阶段 Prompt 模板（可复制）

每个阶段包含：你对 AI 说的话、AI 应交付什么、你如何验收。

P0 — CAN 硬件层

你对 AI 说：

我有一块 PEAK PCAN-USB，推杆用 CAN-FD 通信。
请基于官方 PCANBasic.py，写一个本地 can.py：
- 提供 Message、Bus、CanError，接口模仿 python-can
- 默认 PCAN_USBBUS1，波特率仲裁 1Mbps、数据 5Mbps
- 重要：项目内 import can 指导入本地文件，不要 pip install python-can
请只交付 can.py，并给一段测试收发示例。

AI 应交付：can.py + 初始化/发送/接收示例

你验收：bus = can.Bus() 不报错；能 send 且 recv 不抛异常

P1 — 最小位置控制

你对 AI 说（附上 xlsx）：

附件是 KAI 执行器 32 字节通信协议。
先不要写 GUI。写一个 position_light.py：
1. 用户输入关节 ID
2. 使能：仲裁 ID 0x200+id，数据 00000000000700
3. 位置模式：0x01 + 目标位置 int16 小端 + 时长 ms + 0F00
4. 等待后失能 00000000000400
单位：位置 0.01mm。CAN-FD 帧。

AI 应交付：position_light.py

你验收：输入 ID 和位置后，推杆实际伸出对应长度

P2 — CLI 位置校准

你对 AI 说（附上 xlsx Sheet2/3）：

实现 calib_v3.py 命令行校准，流程：
1. 使能
2. SDO 0x600+id 写 offset=0：2B01200500000000
3. 发 standby 开启上报：2B03180520000000
4. 读 0x480+id 反馈，解析 [2:4] 为位置 int16 小端
5. 用户输入实际位置（0.01mm），计算 offset = 实际 - 反馈
6. SDO 写 offset：2B012005 + offset_hex + 0000
7. 失能
先做 v3 单次采样；通过后再做 v4 加多次采样和重试。

AI 应交付：calib_v3.py，后续 calib_v4.py

你验收：校准后读回位置与用户输入一致（误差 < 0.05mm）

P3 — GUI 上位机

你对 AI 说：

用 Tkinter 写 slider_upper.py 推杆上位机 V5.2：
- 连接 PCAN、扫描关节 ID 1~10
- 校准：复用 calib 六步流程，目标位置 mm，容差 0.05mm
- 移动：闭环到目标，行程按型号 LA5000/2000/400 限制
- 状态面板：按 xlsx Sheet3 解析 0x480 的 32 字节
- CAN 放后台线程，GUI 用 root.after 更新
- 注意 PCAN shutdown 有 bug，断开时尽量复用 bus

AI 应交付：slider_upper.py

你验收：连接、校准、移动、状态刷新、改 ID、力传感器标零均可操作

P4 — 文档与交付

你对 AI 说：

为 can.py、calib_v3.py、slider_upper.py、PCANBasic.py 加中文注释；
写 teach.html 讲解代码结构；
写 slider_upper.spec 用 PyInstaller 打包；
不要改业务逻辑。

AI 应交付：注释源码 + teach.html + slider_upper.spec

你验收：新人读 teach.html 能看懂；pyinstaller slider_upper.spec 能出 exe

每次对话应提供的「上下文包」

优先级材料说明
必附KAI执行器通信32字节协议.xlsx或导出 Sheet2~3 为 Markdown / 截图
必说关节 ID、推杆型号、行程如 ID=1，LA5000，0~90mm
建议附当前相关 .py 文件让 AI 在现有代码上改，而非重写
建议附一次成功报文的 hex例：msg.data.hex() 输出，便于对照偏移
建议附报错全文 + 操作步骤「点连接后报 CAN 初始化失败」比「不能用」有效
可选PCAN-View 抓包仲裁 ID、数据长度、时序

好的 Prompt vs 坏的 Prompt

坏的例子

帮我写个推杆上位机，要能控制电机。

问题：无协议、无硬件、无验收、无分阶段 → AI 只能幻觉协议。

坏的例子

pip install python-can，然后写 CAN 通信。

问题：与本项目本地 can.py 冲突，导入错模块。

好的例子

附件 xlsx Sheet3 定义了 TxPDO4（COB-ID 0x480+NodeID）。
请用 int16 小端解析 byte[2:3] 为 DriveResp_PosReal，
推杆单位 0.00001 m（即 0.01mm 整数）。
先写函数 parse_position(msg.data)，再集成到 GUI。
验收：与 PCAN 抓包数值一致。

好的例子

calib_v3 校准后误差仍大于 1mm。请对照 xlsx：
- SDO 写 offset 索引是否为 0x2005 子索引 0x01
- 写之前是否已发 standby 2B03180520000000
- 接收超时是否因为没清 recv 队列
在现有 calib_v3.py 上最小改动修复，不要重写。

常见翻车与纠偏话术

现象可能原因你对 AI 说
import can 行为不对装了 pip 的 python-can「卸载 python-can，只用项目目录下的 can.py」
位置数值差 100 倍单位搞错（mm vs 0.01mm）「对照 xlsx：推杆位置单位 0.00001m，列出每个字段换算表」
收不到 0x480 报文未开上报 / ID 偏移错「先发 2B03180520000000，仲裁 ID 用 0x480+device_id」
第二次连接失败PCAN shutdown bug「断开时不调用 shutdown，复用 bus 引用，见 slider_upper _init_can」
GUI 点连接卡死主线程阻塞 recv「CAN 操作放 threading.Thread，UI 更新用 root.after(0, ...)」
校准成功但移动偏校准与移动单位不一致「统一 target_position = int(mm * 100)，全项目搜索换算」

协议 Excel 使用说明（仓库不包含原文）

文件名：KAI执行器通信32字节协议.xlsx（请向设备厂商索取，本地自备）

Sheet3 核心字段（TxPDO4，0x480 + NodeID，32 字节）

与 slider_upper._update_status_display 一致，完整定义以 xlsx 为准：

Byte字段类型推杆单位
0-1DriveResp_Statusuint16状态字
2-3DriveResp_PosRealint160.00001 m（代码中 ×100 为 0.01mm 整数）
4-5DriveResp_VelRealint160.0001 m/s（代码中 ×0.1 显示 mm/s）
6-7DriveResp_TorqRealint161 N
8-9DriveResp_Errorcodeuint16故障码，见 Sheet4
10-12Ia, Ib, Icint8×31 A
13Udcint81 V
14-15Idcint160.01 A
16-23Id, Iq, IdRef, IqRefint16×40.01 A
26-27TorqRefint161 N
28-29MOS温度, 电机温度int8×2℃
30-31DataIndexuint16数据索引

Sheet2 常用 SDO（0x600 + NodeID）

功能数据 hex（示例）
offset 清零2B01200500000000
开启上报 standby2B03180520000000
写 offset2B012005 + offset(2B小端) + 0000
改设备 ID2F012001 + new_id(1B) + 000000

控制帧（0x200 + NodeID）

功能数据 hex
使能00000000000700
失能00000000000400
位置模式01 + pos(2B) + duration_ms(2B) + 0F00
